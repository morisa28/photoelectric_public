# new(7) 阶段接续文档 3

## 当前状态

`new(7)` 已从 `new（6）` 的“可运行但混层较重”状态，进入“主链路结构已收口”的版本。

接手时先看下面这些文件：

1. `src/stores/experiment.js`
2. `src/features/experiment/constants.js`
3. `src/features/experiment/helpers.js`
4. `src/composables/useVirtualLabController.js`
5. `src/composables/usePhotoelectricSceneController.js`
6. `src/three-lab/createPhotoelectricLabScene.js`
7. `src/three-lab/scene/*`

## 当前最重要的边界

### 1. 状态单一来源没有变

仍然只有一个全局实验状态源：

1. `src/stores/experiment.js`

抽离出去的模块只负责：

1. 常量
2. 纯计算
3. 状态归一化

不要把业务状态再拆成多个 store。

### 2. 组合式控制器是桥接层，不是新状态层

新增的两个 composable：

1. `useVirtualLabController.js`
2. `usePhotoelectricSceneController.js`

职责是：

1. 组织页面数据
2. 挂生命周期
3. 转发动作

不要在这里重新发明持久状态或复制 store。

### 3. Three 场景拆层后不要再回塞

现在 `src/three-lab/scene/` 已经有明确拆分。

后续如果继续改 3D：

1. 场景搭建进 `buildClassroom.js` 或新增 scene 模块
2. 实验台对象进 `buildDeskAssembly.js` 或新增 objects 模块
3. 模型接入逻辑进 `loadImportedTestbed.js`
4. 粒子与动画不要再塞回 `createPhotoelectricLabScene.js`

## 当前残余风险

1. `VirtualLab.vue` 虽然脚本逻辑已抽离，但模板和样式体量仍然偏大
2. `createPhotoelectricLabScene.js` 已显著缩小，但交互运行时与渲染循环仍在同文件
3. 旧链路 `public/3d/*` 仍然保留，只是当前未进入主路由
4. 共享目录无法稳定写项目内 `dist/`，正式产物构建应优先输出到其他可写目录验证

## 下一轮最合适做什么

下一轮不要跳去做视觉，建议优先做“验证 / 清理”轮次：

1. 补结构检查文档，标记主链路与旧链路
2. 评估 `VirtualLab.vue` 是否继续拆成更小的 UI 子组件
3. 评估 `createPhotoelectricLabScene.js` 是否继续把交互运行时抽成独立模块
4. 如果要做交互优化，也只做交互，不要混结构和质感

## 一句话交接

`new(7)` 现在已经不再是“所有逻辑往大文件里堆”的状态；后续继续开发时，优先守住现有层次，不要把抽离出去的职责重新塞回壳组件和场景主文件。
