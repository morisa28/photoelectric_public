# new(12) 阶段接续文档 8：3D 旋钮输入修补后的接续说明

## 1. 当前阶段结论

当前项目已经从“3D 旋钮拖不动、左键输入冲突、排查信息不足”的状态，推进到“当前默认视角下可以稳定操作”的状态。

本轮结束后，已经可以确认：

1. 普通左键只负责按钮点击和旋钮拖拽
2. `Alt + 左键` 保留 orbit 视角拖动
3. 中键和滚轮不受 `Alt` 限制
4. 量程旋钮和调零旋钮在当前默认视图下都能拖动生效
5. 诊断面板可以显示当前悬停到的控制面

所以，下一轮不应该再回去重复修“左键冲突”这类表层问题，而是应该继续处理更底层、更稳定性的事情。

---

## 2. 当前实现的真实形态

后续接手时必须先接受一个事实：

当前旋钮交互并不是“纯 3D 透明面命中已经完全修好”的版本，而是：

1. 先尝试走 runtime 内部 raycaster 命中透明面
2. 打不到时，用屏幕热点做兜底

也就是说，本轮交付的是一个“已经可用、但仍带技术债说明”的版本。

不要误判成：

1. 透明面事件已经彻底修好
2. runtime 射线命中已经完全可信
3. 以后怎么转相机都一定成立

这三点目前都还不能下这个结论。

---

## 3. 后续最应该继续做的事

### 3.1 优先查清透明面 / 旋钮射线命中为什么不稳定

这是后续最值得继续追的主问题。

因为当前已经验证过：

1. `range_plane`
2. `zero_plane`
3. `Knob_CurrentRange`
4. `Knob_ZeroAdjust`

这些对象在场景里“存在”，但在 runtime 实际交互时并不容易被命中。

下一轮建议重点排查：

1. 这些对象是否被场景层级、可见性、辅助对象或克隆逻辑影响
2. `page.raycastWithClones(...)` 为什么大多返回 `Room`
3. 当前导出的 `.splinecode` 是否让这几个对象在命中层里发生了退化
4. 是否存在材质、helper、包围盒或对象类型导致的射线筛选偏差

如果这个根因查清并修掉，热点兜底就可以弱化甚至移除。

### 3.2 如果短期不处理根因，就把热点兜底继续做稳

如果下一轮不准备继续下钻 runtime 底层，那就应该把热点兜底方案做得更稳，而不是保持在“刚好能用”。

可继续做的方向：

1. 把热点参数抽成独立常量或配置
2. 给热点增加调试可视化
3. 允许诊断模式直接显示热点中心和半径
4. 在缩放或窗口尺寸变化时自动复算

当前热点参数在：

1. `centerXRatio`
2. `centerYRatio`
3. `radiusX`
4. `radiusY`

只要默认视图变了，就应该重新校准。

## 5. 建议的后续工作顺序

建议按下面顺序继续，不要乱序：

1. 先确认当前热点兜底方案在默认视图下仍然稳定
2. 再确认 `Alt + 左键 / 普通左键 / 中键 / 滚轮` 分流没有回归
3. 然后决定是否继续深挖 runtime 射线命中根因
4. 如果要深挖，就优先做诊断和最小复现，不要先改业务逻辑
5. 如果暂时不深挖，就把热点参数调试工具补齐

原因：

1. 当前版本已经可用，先别把可用状态打掉
2. 输入分流是用户直接感知的交互，回归代价最高
3. runtime 命中问题更像底层行为问题，需要单独隔离验证

## 6. 推荐继续使用的验证方式

下一轮继续开发时，建议保留下面几类验证：

### 6.1 构建验证

```bash
npm run build
```

这是最低限度，不通过就不要继续看交互。

### 6.2 浏览器程序化交互验证

本轮已经用 Playwright 验证过：

1. `hoveredSurface` 能显示 `range_plane` / `zero_plane`
2. 拖动后 `currentRange` 会改变
3. 拖动后 `zeroAdjust` 和 `currentPhotocurrentMicroamp` 会改变
4. 中键、普通左键、`Alt + 左键` 的 orbit 状态分流正确

所以下一轮如果继续改输入逻辑，仍然建议优先写这类程序化验证，而不是只靠肉眼试。

### 6.3 诊断模式人工验证

手测时建议直接打开：

```text
/3d-view?diagnostic=1&wait=2200&zoom=1.12
```

重点看：

1. `轨道控制`
2. `悬停透明面`
3. 拖拽步数
4. 电流 / 电压屏的状态变化

## 7. 当前已知风险，不要忘

后续接手时最容易忽略的风险有三个：

1. 热点兜底是和当前默认正视角绑定的
2. 如果后续改相机、缩放、构图或模型位置，热点参数会漂
3. `projectObjectToScreen()` 现在还不足以直接替代热点方案

因此，下一轮如果改了视觉构图，一定要顺手重新验证：

1. `hoveredSurface`
2. 量程拖动后的 `currentRange`
3. 调零拖动后的 `zeroAdjust`

不要只看“看起来位置差不多”。

## 9. 本阶段结论

当前阶段已经完成的不是“3D 输入彻底完美”，而是：

1. 把最影响可用性的交互问题先收住
2. 补齐继续排查所需的诊断信息
3. 明确识别出真正还没彻底解决的底层问题

后续最合理的推进方向是二选一：

1. 继续深挖 runtime 命中根因，争取去掉热点兜底
2. 接受热点兜底为阶段性方案，把它做成更稳的可维护配置层

不管选哪条路，都不要再回到“重新改一遍左键分流”的低层重复劳动里。
